# [MNG-01] 文書管理・ドキュメント台帳 (Document Management & Ledger) - ゆうぞら (Yuzora)

本ドキュメントは、「ゆうぞら (Yuzora)」プロジェクトにおいて作成・維持されるすべてのドキュメントの台帳であり、ドキュメントごとの目的、担当、および設計ドキュメントのオーバーラップ時におけるすみ分け方針を定義します。

---

## 1. 文書管理方針 (Document Management Policy)

ゆうぞらプロジェクトでは、 **「ドキュメントはコードと同様の価値を持つ重要な成果物である」** という方針のもと、極めて厳格に文書管理を行います。
* **ドキュメントの最新性担保**: コードの修正や仕様の変更が発生した際は、実装の前に（または並行して）必ず関連ドキュメントを更新し、設計とコードの乖離（ドキュメントの腐敗）を完全に防止します。
* **Gitによる一元管理**: すべてのドキュメントは Markdown 形式で作成し、ソースコードと同一 of Git リポジトリ（`docs/` ディレクトリ）で管理することで、世代管理と変更履歴のトレーサビリティを担保します。

### 1.1 文書管理番号の設計方針と管理策 (Numbering Policy & Controls)

新規ドキュメントの追加や再構成に伴う「管理番号・ファイル名の書き換えインパクト（リンク切れ等）」を最小化し、変更に強いドキュメント構成を維持するため、以下の付与方針および管理策を定めます。

#### A. 分類プレフィックスを用いたドメイン分割（方針）
ドキュメントを一律の通し番号ではなく、機能・性質ごとに定義した以下の**分類プレフィックス（2〜3桁のアルファベット）＋2桁の連番（01から開始）**の構成とします。
* **`MNG` (Management)**: 文書管理台帳、開発プロセス定義書などの管理・プロセスドキュメント。
* **`REQ` (Requirements)**: 要求定義書、要件定義書、機能一覧などの要求・要件関連ドキュメント。
* **`DSN` (Design)**: 基本設計書、詳細設計書などの技術設計ドキュメント。
* **`USR` (User Documentation)**: 一般ユーザーやシステム管理者向けの操作マニュアルやガイドブック。
* **`ADR` (Architecture Decision Records)**: 技術的な意思決定とその背景、影響を記録するADR。

> [!TIP]
> **変更耐性のメリット**: 
> 各分類で番号体系が独立しているため、例えば要求・要件定義領域で新規ドキュメント（例: 機能一覧）が追加されても、管理（`MNG-`）や設計（`DSN-`）、マニュアル（`USR-`）といった他ドメインの文書番号・ファイル名には一切影響を及ぼしません。

#### B. 文書管理策 (Document Controls)
1. **登録管理制（発行・廃止コントロール）**:
   * 新たなドキュメントを作成・削除する際は、必ず本ドキュメント（MNG-01）の「2. ドキュメント台帳」に事前登録し、定義されたプレフィックスに基づく一意の管理番号を発行します。
2. **リンク整合性維持（監査コントロール）**:
   * ファイル名の変更や新規ドキュメント追加が行われた際は、ドキュメント間での参照リンク（相対リンク）が正常に維持されているかを全体スキャンし、デッドリンクを完全に防止します。
3. **重複排除（二重メンテナンス防止）**:
   * 同一仕様の重複記述を排除し、情報の一元化（Single Source of Truth）を徹底します（[3. 設計ドキュメント間のすみ分けと分掌] を遵守）。仕様変更時のドキュメント更新箇所が最小化され、管理番号の整合性維持が容易になります。

## 3. 設計ドキュメント間のすみ分けと分掌 (Demarcation & Overlap Management)

要求、要件、基本設計(HLD)、詳細設計(LLD)、および TOGAF EA ドメインの記述内容は一部オーバーラップすることがあります。その際の記述のすみ分け、および競合時の分掌ルールを以下のように規定します。

### 3.1 HLD（基本設計）と LLD（詳細設計）のすみ分け
* **原則**: 
  * HLDは **「何が（What）」定義されているか（論理・概念）** に終始します。
  * LLDは **「どのように（How）実装するか（物理・具象）」** を詳細に規定します。
* **オーバーラップの排除例**:
  * **データ構造**:
    * HLD：データが `localStorage` に保存されること、保存するエンティティの項目名（テーマ、フォントサイズ等）を論理的に記述。
    * LLD：具体的なキー名（`yuzora_config`）、値のJSONシリアライズスキーマ、データ型を物理レベルで記述。
  * **デザイン・スタイル**:
    * HLD：カラーテーマのカラー定義（和紙、明、暗、漆黒）やフォント選定（Noto Serif JP）などの共通デザイン原則を記述。
    * LLD：CSS変数の具体的なカラーコード（`#f5eedc`等）、フォントサイズ（`.size-md: 1.25rem`等）のクラスマッピング値を記述。

### 3.2 要求・要件と設計（HLD/LLD）のすみ分け
* **原則**:
  * 要求・要件は **「システムが提供すべき仕様・制約（What the system must do）」** であり、実装技術やモジュール構造からは独立しています。
  * 設計は **「その要件を技術的にどう解決するか（How the system solves it）」** を定義します。
  * **分掌の競合解決ルール**:
  * 実装や検証の過程で要件定義（SRD）と設計（HLD/LLD）の間に矛盾が発覚した場合、 **「要件定義（SRD）が上位」** となります。
  * 設計側の都合で要件を変更する場合は、まず [REQ-03-system_requirements.md](/docs/REQ-03-system_requirements.md) の変更履歴（要件変更プロセス）を経て要件を改定したのち、設計書（HLD/LLD）をそれに追従させます。

### 3.3 TOGAF EA ドメイン間のすみ分け
TOGAF EA の4ドメイン（BA, AA, DA, TA）に沿った記述のオーバーラップは、以下のマトリクスに基づき、ドキュメントの目的ごとに分担（分掌）します。

| EAドメイン | 要求定義 (URD) での表現 | 要件定義 (SRD) での表現 | 基本設計 (HLD) での表現 | 詳細設計 (LLD) での表現 |
| :--- | :--- | :--- | :--- | :--- |
| **ビジネス (BA)** | ユーザーの読書プロセスと目標、必須対象作品、ロードマップ。 | （記述しない） | （記述しない） | （記述しない） |
| **アプリケーション (AA)** | （記述しない） | システムが提供する機能要件（ファイル読み込み、パース、しおり）。 | クライアントサイドSPA（View, Controller, Storage）の論理構成とデータフローシーケンス。 | 各関数のインターフェース仕様（引数、戻り値、処理フロー）。 |
| **データ (DA)** | （記述しない） | デコード対応、しおりの保存要件などの要件定義。 | テキスト/XHTMLの論理的解釈、しおりの論理情報項目。 | 正規表現置換ルール、LocalStorage物理JSONスキーマ。 |
| **技術 (TA)** | 動作ブラウザ環境要件。 | Web標準技術（HTML5, Vanilla CSS, JS）、およびフォント要件。 | クライアントサイド技術スタック選定、CSSカスタムプロパティ変数定義。 | 具体的なCSS変数マッピング値、フォントローディング実装詳細。 |
